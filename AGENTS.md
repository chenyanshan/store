# AGENTS.md

## 项目目标

将当前仓库构建成一个中文案例数据库网站，并部署到 Cloudflare Workers。

这不是一个通用营销站，而是一个围绕零售创业案例构建的结构化内容数据库，并逐步扩展为可持续使用的学习工具。

## 产品方向

- 网站类型：案例数据库
- 目标用户：浏览、对比、学习零售创业案例的中文用户
- 核心价值：帮助用户按分类和结果快速找到案例，查看详情，并在学习模式中持续复盘和积累

## 不可妥协的优先级

在此仓库中工作的 agents 应优先保障以下能力：

1. 分类筛选
2. 案例详情展示
3. 按 `success` / `failure` / `warning` 聚合展示
4. 学习模式（顺序浏览、收藏、进度记录）
5. 优秀的移动端体验

如果出现取舍，优先保护以上五项。

## 技术方向

使用能在 Cloudflare Workers 上稳定运行的最简单方案。

默认技术栈：

- 静态站点，使用原生 HTML、CSS、JavaScript
- 构建时将 `data.js` 转换为 JSON
- 使用 `wrangler` 配置 Cloudflare Workers Static Assets

默认不引入 React、Next.js、Astro、CMS 或 SSR。

如果用户明确要求“学习功能”中的跨设备同步、收藏、进度存储，则允许在 Cloudflare Workers 上增加服务端 API 和数据库。

学习功能默认技术选择：

- 前端仍保持原生 HTML、CSS、JavaScript
- 服务端接口仍由 Cloudflare Workers 提供
- 学习状态存储优先使用 Cloudflare D1
- 浏览器本地持久化优先使用 `localStorage`

当前 Cloudflare 的官方方向更偏向使用 Workers Static Assets 承载新的静态或全栈项目，实现时应尽量保持一致。

## 学习功能规划

### 目标

学习功能不是传统登录系统，而是“匿名学习状态系统”。

用户可以：

- 进入学习模式，一条一条顺序学习案例
- 在学习模式中点击上一条、下一条
- 收藏案例
- 记录学习进度
- 刷新浏览器后继续保留当前学习 ID 和进度
- 在手机端和未来 App 端通过同一个学习 ID 恢复同一份学习状态

### 页面与交互

学习功能默认新增一个独立页面：

- `/study`

学习模式的默认交互应包括：

1. 输入学习 ID 并开始/继续学习
2. 展示当前案例内容
3. 上一条 / 下一条
4. 收藏（在详情弹窗中收藏或取消收藏）
5. 仅看收藏视图
6. 显示当前学习位置，例如“第 12 条 / 共 80 条”
7. 支持退出当前学习 ID

第一版推荐使用“单案例沉浸式浏览”模式，而不是列表中无限展开。

### 关键产品判断

学习功能不推荐做成很多独立详情子页面一路翻页。

第一版更合理的方案是：

- 只保留一个学习主页面 `/study`
- 页面内部展示当前案例
- 用“上一条 / 下一条”切换
- 用服务端保存当前位置

这样做的原因：

- URL 结构更简单
- 刷新恢复更容易
- 更适合移动端单手操作
- 更适合未来 App 复用同一套学习状态接口

如果后续需要“分享当前学习中的某一条案例”，可以再增加：

- `/study?case=xxx`

但这不是第一版刚需。

### 学习序列规则

默认情况下，学习模式应以稳定顺序浏览案例。

推荐顺序：

- 默认按 `data.js` 中的原始顺序
- 如果用户从筛选结果进入学习模式，可基于当前筛选结果生成一份学习清单快照

不要在第一版默认做复杂的智能推荐、随机重排或算法排序。

### 学习模式页面结构

`/study` 页面第一版推荐拆成以下区域：

1. 学习身份区
   - 输入学习 ID
   - 继续学习 / 新建学习
   - 退出当前设备
2. 学习进度区
   - 当前第几条
   - 总数
   - 最近同步状态
3. 案例内容区
   - 标题
   - 结果标签
   - 分类 / 地点 / 投资等元信息
   - 摘要
   - 关键要点
   - 完整详情
4. 学习操作区
   - 上一条
   - 下一条
   - 收藏（在详情弹窗中执行）
   - 取消收藏（在详情弹窗中执行）
5. 辅助区
   - 进入收藏列表
   - 返回案例库
   - 恢复码说明（如果启用）

第一版不要在学习页同时塞入复杂筛选面板、巨型侧栏或密集分析图表。

### 学习模式交互流程

推荐默认流程：

1. 用户进入 `/study`
2. 如果本地有学习 ID，自动尝试恢复
3. 如果本地没有学习身份，提示输入学习 ID 与恢复码
4. 服务端返回当前进度
5. 前端直接展示当前案例
6. 用户点击“下一条”时，前端先切换，再异步写入云端
7. 写入成功后更新时间戳
8. 写入失败时提示“本地已切换，云端同步失败，可稍后重试”

这意味着：

- 阅读体验优先
- 同步应尽量异步，不要阻塞翻页
- 但关键状态要有失败提示

### 收藏页形态

第一版不建议单独做复杂的“收藏中心”子系统。

推荐最小方案：

- 在 `/study` 内支持切换到“仅看收藏”
- 收藏视图采用卡片列表，卡片入口为“查看详情”
- 收藏/取消收藏统一在详情弹窗内处理

不要在第一版新增大量二级路由和复杂列表管理逻辑。

### 收藏与进度

第一版学习功能至少要支持两类状态：

- 学习进度
- 收藏列表

推荐的最小状态模型：

- 当前学习 ID
- 当前学习到的案例 ID
- 当前学习序号
- 收藏的案例 ID 集合
- 最近学习时间

当前实现要求：

- 学习进度游标仅基于“全部案例序列”
- 收藏视图不应复用或覆盖主学习进度游标

如需更细粒度，可再增加：

- `studying` / `done` 状态
- 学习历史
- 复习队列

## 学习身份模型

### 基本要求

用户无需注册账号，也无需邮箱密码登录。

第一版可采用：

- 用户手动输入“学习 ID + 恢复码”
- 浏览器用 `localStorage` 记住当前学习 ID 与恢复码
- Workers API 用这两项在服务端查询和写入进度

### 风险与建议

仅靠“学习 ID”本身就同步跨设备，是可实现的，但安全性较弱。

风险：

- 任何知道该学习 ID 的人都可能读取或覆盖这份学习进度
- 如果学习 ID 太简单，会被误撞或恶意枚举

因此推荐的默认方案是：

- `学习 ID + 恢复码/口令`

这仍然不算传统登录系统，因为不需要账号密码注册流程，但安全性明显更合理。

当前代码基线已按此方案落地。若要改为“只输入一个 ID”，需要同步修改前端校验与 `worker.js` 接口校验逻辑。

### 学习 ID 规则建议

学习 ID 不应过短，也不应允许用户随意输入极弱字符串。

推荐要求：

- 长度至少 8 位
- 仅允许字母、数字、短横线、下划线
- 不区分大小写或统一转小写

当前不启用前端自动生成学习 ID。如未来要启用，推荐生成：

- 可读性较好的短 ID
- 例如 10 到 14 位

如果启用恢复码/口令：

- 当前实现为 4 位数字恢复码
- 前后端均强校验：`^\d{4}$`
- 浏览器本地缓存当前学习 ID 与恢复码

### 学习 ID 创建策略

当前实现采用：

1. 用户手动输入学习 ID
2. 前端不提供“系统生成学习 ID”

如需恢复自动生成功能，必须同时更新：

- 前端输入与事件绑定
- 文档和用户提示文案

### 退出与注销定义

必须区分两个动作：

1. 退出当前学习 ID
   - 只清除当前设备浏览器保存的学习 ID
   - 云端进度不删除
2. 永久删除学习数据
   - 删除该学习 ID 在服务端的进度和收藏

如果用户说“注销”，默认解释为“退出当前设备”；不要默认删除云端数据。

## Cloudflare 功能规划

如果要实现匿名学习、收藏、跨设备同步，Cloudflare 侧需要补充以下能力：

### 必需能力

1. Cloudflare Workers API
   - 负责读写学习进度与收藏
2. Cloudflare D1
   - 作为学习数据的主存储
3. 浏览器本地存储
   - 使用 `localStorage` 保存当前学习 ID，保证刷新后仍能恢复

### `wrangler` 侧新增内容

如果开始实现学习功能，`wrangler` 配置默认需要补充：

- D1 数据库绑定
- 可选的环境区分（`dev` / `production`）

推荐命名：

- D1 binding: `DB`

当前代码兼容：

- 首选 `env.DB`
- 兼容 `env.retail_case_database`（用于历史配置过渡）

第一版不必同时引入太多 bindings。

### 为什么优先选 D1

D1 更适合这类“按学习 ID 查询、更新当前进度、收藏案例、按时间排序”的结构化数据。

对于第一版学习功能：

- D1 适合做主数据库
- KV 不适合做核心学习进度主存储
- Durable Objects 也不是第一版刚需

### 不推荐作为第一版核心方案的能力

- Workers KV
  - 更适合高读低写、缓存、配置数据
  - 不建议把学习进度主数据只放在 KV
- Durable Objects
  - 更适合多客户端强协调、实时协作、聊天室、在线对战等场景
  - 当前学习功能第一版不需要这么重的协调能力
- R2
  - 当前不需要大文件对象存储

### 可选增强能力

如站点公开访问量较大，可选增加：

- Turnstile 或其他轻量防滥用方案
- API 写接口的简单限流
- 运营统计与日志

- 服务端失败时需要回滚或提示重试

不要为了第一版引入过重的同步冲突解决系统。

## API 设计建议

学习功能的第一版接口应保持简单。

推荐至少包含：

- `POST /api/study/open`
  - 使用 `learning_id + secret` 打开或创建学习会话
- `GET /api/study/state`
  - 获取当前学习进度和收藏（支持 header 或 query 传凭证）
- `POST /api/study/progress`
  - 更新当前案例位置
- `POST /api/study/favorites/toggle`
  - 收藏或取消收藏
- `POST /api/study/logout`
  - 退出当前设备上的学习 ID

如果以后做 App，同样复用这一组接口。

### API 契约原则

所有学习接口默认使用 JSON。

推荐原则：

- 请求体和响应体字段命名统一
- `case_id`、`learning_id` 等字段名保持稳定
- 不要一会儿返回数组、一会儿返回对象包裹数组

推荐响应结构：

- `ok`
- `data`
- `error`

例如：

- 成功：`{ ok: true, data: { ... } }`
- 失败：`{ ok: false, error: { code, message } }`

### 第一版接口职责边界

推荐按下述方式收口：

- `open`
  - 负责创建或恢复学习会话
- `state`
  - 负责读取完整学习状态
- `progress`
  - 只更新当前学习位置和最近时间
- `favorites/toggle`
  - 只处理收藏
- `logout`
  - 只清除当前设备使用的学习上下文，不删除云端

不要把“创建、恢复、更新进度、收藏、删除数据”塞进一个万能接口。

### App 兼容要求

如果未来要做手机 App，当前 API 设计必须遵守：

- 无浏览器特定语义
- 不依赖 DOM
- 不依赖 Cookie 才能工作
- 仅依赖明确传入的 `learning_id` 和必要凭证

这样 Web 和 App 都能复用同一套后端。

- 导出或备份学习记录
- App 端接入

## 实施注意事项

开始写学习功能代码前，agents 应先确认以下决策是否已经定稿：

1. 是否只允许“学习 ID”，还是启用“学习 ID + 恢复码”
2. `/study` 是否只做单页沉浸式浏览
3. 第一版是否支持“仅看收藏”
4. 学习顺序是否固定为原始 `data.js` 顺序
5. 是否允许从筛选结果生成学习队列

如果这些问题未定，默认采用本文档中的推荐方案，不要临时自由发挥。

## 实现边界

- 依赖保持最少
- 优先使用原生 JavaScript 和简单目录结构
- 优先采用静态资源加轻量前端逻辑，不要引入不必要的后端复杂度
- 当用户明确要求学习同步功能时，允许引入 D1 和 Workers API
- 除非用户明确要求，否则不要引入完整账号体系、邮箱登录、OAuth 或第三方鉴权
- 在核心数据库体验完成之前，不要过度设计 SEO 或 CMS 工作流
- 不要重命名核心数据字段；如果必须调整，先提供兼容层
- 不要把学习功能第一版做成重社交产品、实时协作产品或复杂刷题系统

## 建议实施顺序

agents 一般应按以下顺序推进：

1. 搭建最小静态站点骨架
2. 增加构建步骤，将 `data.js` 转成 JSON
3. 实现案例列表页和结果聚合
4. 实现分类筛选
5. 实现详情弹窗与知识总结页
6. 优化移动端布局和交互
7. 增加 `wrangler` 配置和部署流程
8. 设计学习模式页
9. 增加学习 ID、本地持久化和 D1 数据模型
10. 增加学习进度与收藏 API
11. 实现跨设备同步与退出机制

## 完成标准

第一版可用版本至少应满足以下条件：

- 项目可以通过 Cloudflare 工具链在本地运行
- 项目可以通过 `wrangler deploy` 部署
- 案例数据来自构建生成的 JSON，且源头是 `data.js`
- 用户可以按分类筛选案例
- 用户可以浏览成功、失败、警示三类聚合结果
- 用户可以查看任意案例的详情弹窗
- 网站包含知识总结页
- 移动端浏览体验可靠

学习功能第一版完成标准至少应满足以下条件：

- 用户可以进入 `/study`
- 用户可以输入学习 ID + 4 位恢复码并继续学习
- 浏览器刷新后仍能保留当前学习 ID
- 用户可以顺序查看上一条 / 下一条案例
- 用户可以在详情弹窗中收藏和取消收藏案例
- 学习模式“仅看收藏”卡片列表可查看详情，并在详情弹窗中取消收藏
- 同一个学习 ID 可以在另一台设备恢复到相近的学习状态
- 用户可以退出当前设备保存的学习 ID
- 如启用恢复码/口令，服务端必须校验

## 决策偏好

当存在多个可行方案时，优先选择以下特征更强的方案：

1. 更简单
2. 更容易部署到 Cloudflare Workers
3. 更适合围绕单一本地数据集维护
4. 后续扩展时不需要重写整套技术栈
