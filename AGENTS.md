# AGENTS.md

## 项目目标

将当前仓库构建成一个中文案例数据库网站，并部署到 Cloudflare Workers。

这不是一个通用营销站，而是一个围绕零售创业案例构建的结构化内容数据库。

## 产品方向

- 网站类型：案例数据库
- 目标用户：浏览、对比、学习零售创业案例的中文用户
- 核心价值：帮助用户按分类和结果快速找到案例，并进入详情页阅读全文

## 不可妥协的优先级

在此仓库中工作的 agents 应优先保障以下能力：

1. 分类筛选
2. 案例详情页
3. 按 `success` / `failure` / `warning` 聚合展示
4. 优秀的移动端体验

如果出现取舍，优先保护以上四项。

## 技术方向

使用能在 Cloudflare Workers 上稳定运行的最简单方案。

默认技术栈：

- 静态站点，使用原生 HTML、CSS、JavaScript
- 构建时将 `data.js` 转换为 JSON
- 使用 `wrangler` 配置 Cloudflare Workers Static Assets

除非用户明确要求，否则不要默认引入 React、Next.js、Astro、数据库、CMS 或 SSR。

当前 Cloudflare 的官方方向更偏向使用 Workers Static Assets 承载新的静态或全栈项目，实现时应尽量保持一致。

## 数据真源

当前仓库的核心数据文件是 [`data.js`](/Users/chenyanshan/Documents/vibecoding/store/data.js)。

规则如下：

- `data.js` 是案例内容的唯一真源
- 除非用户明确要求，否则不要手动把源数据改写成另一套内容维护格式
- 网站使用的数据产物应在构建阶段由 `data.js` 转换生成
- 除非用户明确要求清洗数据，否则保留所有现有记录和字段

当前数据结构：

- 根变量名：`cases`
- 记录总数：约 95 条
- 标准结果值：`success`、`failure`、`warning`

已知字段：

- `id`
- `title`
- `category`
- `result`
- `resultText`
- `location`
- `investment`
- `monthlyProfit`
- `monthlyLoss`
- `businessTime`
- `summary`
- `background`
- `problem`
- `process`
- `resultDetail`
- `reflection`
- `keyPoints`
- `tags`

## 数据处理规则

- 保持 `id` 稳定且唯一
- 保留中文原文，不要静默改写数据内容
- 空字符串应视为缺失值，而不是有业务含义的值
- `keyPoints`、`tags` 这类数组字段必须保持数组结构
- 如果需要新增派生字段，应在构建产物中生成，不要直接修改原始数据

## 默认信息架构

除非用户另有说明，否则网站默认应围绕以下页面和视图构建：

- 首页/列表页，展示全部案例
- `category` 和 `result` 的筛选控件
- 成功、失败、警示三类结果的聚合统计
- 单个案例的详情页路由

推荐路由模型：

- 使用简单的 SPA 风格路由，例如 `/cases/:id`
- 正确配置 Workers 静态资源回退，确保详情页被直接访问或刷新时仍然可用

## 体验规则

- 必须采用移动端优先设计
- 网站内容语言为中文
- 优先保证信息可读性，而不是装饰性
- 筛选条件和结果聚合应尽早出现在页面上方
- 详情页要清晰呈现摘要、关键点和完整案例上下文
- `success`、`failure`、`warning` 必须有明确的视觉区分

## 实现边界

- 依赖保持最少
- 优先使用原生 JavaScript 和简单目录结构
- 优先采用静态资源加轻量前端逻辑，不要引入不必要的后端复杂度
- 除非用户要求，否则不要引入外部 API、鉴权或数据库
- 在核心数据库体验完成之前，不要过度设计 SEO 或 CMS 工作流
- 不要重命名核心数据字段；如果必须调整，先提供兼容层

## 建议实施顺序

agents 一般应按以下顺序推进：

1. 搭建最小静态站点骨架
2. 增加构建步骤，将 `data.js` 转成 JSON
3. 实现案例列表页和结果聚合
4. 实现分类筛选
5. 实现案例详情路由
6. 优化移动端布局和交互
7. 增加 `wrangler` 配置和部署流程

## 完成标准

第一版可用版本至少应满足以下条件：

- 项目可以通过 Cloudflare 工具链在本地运行
- 项目可以通过 `wrangler deploy` 部署
- 案例数据来自构建生成的 JSON，且源头是 `data.js`
- 用户可以按分类筛选案例
- 用户可以浏览成功、失败、警示三类聚合结果
- 用户可以进入任意案例的独立详情页
- 详情页路由支持直接访问和刷新
- 移动端浏览体验可靠

## 决策偏好

当存在多个可行方案时，优先选择以下特征更强的方案：

1. 更简单
2. 更容易部署到 Cloudflare Workers
3. 更适合围绕单一本地数据集维护
4. 后续扩展时不需要重写整套技术栈
